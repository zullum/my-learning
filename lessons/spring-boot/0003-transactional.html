@Transactional — How Transactions Really Work

The most misunderstood annotation in Spring Boot — and the most dangerous when misused

🎯 Mission: LabelRadar writes to MySQL 5.7 — labels, users, submissions. When multiple writes happen together (e.g. create a submission + update a counter), transactions protect you from partial failures. This lesson shows you exactly how @Transactional works — and the gotchas that bite everyone.

What Is a Transaction?

A transaction is a group of database operations that either all succeed or all get rolled back — as if none of them happened.

💡 Analogy: Transferring money between bank accounts. You debit one account and credit another. If the credit fails after the debit, you want the debit rolled back too. That's a transaction — all or nothing.

How @Transactional Works — Step by Step

When you annotate a method with @Transactional, Spring wraps it in a proxy (same mechanism as auto-configuration). Here's what happens at runtime:

📥
1. Proxy intercepts the method call
Spring's AOP proxy catches the call before your code runs
↓
🔗
2. BEGIN TRANSACTION on the DB connection
Gets a connection from HikariCP, calls setAutoCommit(false)
↓
⚙️
3. Your method runs
All repository calls use this same connection/transaction
↓
✅
4a. No exception → COMMIT
All changes are written to MySQL permanently
↓
💥
4b. RuntimeException thrown → ROLLBACK
All changes from this transaction are discarded
@Service
public class SubmissionService {

    @Transactional  // wraps the entire method in one transaction
    public void submitLabel(Long labelId, Long userId) {
        Label label = labelRepository.findById(labelId).orElseThrow();
        label.setSubmissionCount(label.getSubmissionCount() + 1);
        labelRepository.save(label);           // write 1

        Submission sub = new Submission(labelId, userId);
        submissionRepository.save(sub);        // write 2

        // If ANYTHING above throws RuntimeException → both writes rolled back
    }
}

Where to Put @Transactional — The Rule

Always put @Transactional on the Service layer, not on the Repository or Controller. The Service is where business logic lives — and a transaction should span exactly one business operation.

// ✅ CORRECT — on the Service
@Service
public class LabelService {
    @Transactional
    public void approveLabelAndNotify(Long id) { ... }
}

// ❌ WRONG — on the Repository (Spring Data already handles this)
@Repository
public interface LabelRepository extends JpaRepository<Label, Long> {
    @Transactional  // redundant — JpaRepository methods are already transactional
    List<Label> findByGenre(String genre);
}

Read-Only Transactions — A Free Optimisation

For methods that only read data, use @Transactional(readOnly = true). Hibernate skips dirty-checking (comparing every entity to its original state), which is faster and uses less memory.

@Transactional(readOnly = true)  // tells Hibernate: no writes, skip dirty checking
public List<Label> getTopLabels() {
    return labelRepository.findTop10ByOrderBySubmissionCountDesc();
}

The Gotchas That Bite Everyone

⚠️ Gotcha 1 — Self-invocation doesn't work

Because @Transactional uses a proxy, calling a @Transactional method from within the same class bypasses the proxy entirely — no transaction is started.

@Service
public class LabelService {

    public void doSomething() {
        saveLabel();  // ❌ calls this.saveLabel() directly — proxy bypassed, NO transaction!
    }

    @Transactional
    public void saveLabel() { ... }
}

// Fix: inject the service into itself, or restructure into two classes
⚠️ Gotcha 2 — Checked exceptions don't trigger rollback by default

Spring only rolls back on RuntimeException (unchecked) by default. A checked exception (e.g. IOException) will commit the transaction unless you tell it otherwise.

// ❌ IOException won't trigger rollback — transaction commits!
@Transactional
public void importLabels() throws IOException {
    labelRepository.save(label);
    throw new IOException("file not found"); // commits anyway!
}

// ✅ Fix: declare rollbackFor
@Transactional(rollbackFor = Exception.class)
public void importLabels() throws IOException { ... }
⚠️ Gotcha 3 — MySQL 5.7 + InnoDB only

Transactions only work on InnoDB tables. If any table in LabelRadar was created with MyISAM engine (older default), @Transactional will silently have no effect on it. Always verify: SHOW TABLE STATUS WHERE Name='your_table' — Engine should be InnoDB.

🔍 LabelRadar Pattern — Service with Transactions

@Service
@Transactional(readOnly = true)  // default for all methods: read-only
public class LabelService {

    // read — inherits readOnly=true from class level
    public List<Label> findByGenre(String genre) {
        return labelRepository.findByGenre(genre);
    }

    // write — overrides to readOnly=false
    @Transactional
    public Label createLabel(LabelDto dto) {
        Label label = new Label(dto.getName(), dto.getGenre());
        return labelRepository.save(label);
    }

    // write with rollback on any exception
    @Transactional(rollbackFor = Exception.class)
    public void importLabels(List<LabelDto> dtos) throws IOException {
        dtos.forEach(dto -> labelRepository.save(new Label(dto)));
    }
}
