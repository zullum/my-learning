How JpaRepository Works Under the Hood

From your interface to an SQL query — and back to your LabelRadar entities

🎯 Mission: LabelRadar uses Spring Data JPA to talk to MySQL 5.7. After this lesson you'll know exactly what happens between writing labelRepository.findById(id) and getting a row back from the database.

The Big Picture — Layers

Spring Boot apps using JPA typically have this stack. Every layer has one job:

🌐 Controller HTTP in → calls Service → HTTP out
↓
⚙️ Service Business logic, @Transactional lives here
↓
📦 Repository Your interface — extends JpaRepository
↓
🔧 Spring Data JPA Generates implementations at startup using Hibernate
↓
🗄️ MySQL 5.7 Actual SQL queries via JDBC connection pool (HikariCP)

What is JpaRepository — Really?

When you write this:

public interface LabelRepository extends JpaRepository<Label, Long> {
}

You're declaring an interface with zero implementation code. Yet you can immediately call save(), findById(), findAll(), deleteById()... How?

1
JpaRepository inherits from a chain of interfaces

Each level adds more methods:

JpaRepository<T, ID>
  └─ PagingAndSortingRepository<T, ID>   // findAll(Sort), findAll(Pageable)
       └─ CrudRepository<T, ID>            // save, findById, findAll, delete, count
            └─ Repository<T, ID>             // marker interface only
2
At startup, Spring generates a real implementation

Spring Data scans for all interfaces extending Repository, then creates a SimpleJpaRepository proxy for each one — automatically, as a Spring bean.

// You never write this — Spring generates it behind the scenes:
public class SimpleJpaRepository<T, ID> implements JpaRepository<T, ID> {

    private final EntityManager em;

    public T findById(ID id) {
        return em.find(entityClass, id); // translates to: SELECT * FROM label WHERE id=?
    }

    public T save(T entity) {
        if (isNew(entity)) em.persist(entity);  // INSERT
        else em.merge(entity);                 // UPDATE
        return entity;
    }
}
3
EntityManager talks to Hibernate, Hibernate talks to MySQL

The EntityManager is the JPA API. Hibernate is the implementation. It translates your object operations into SQL and sends them through HikariCP to MySQL 5.7.

Derived Query Methods — The Real Power

Spring Data reads your method names and generates SQL automatically:

public interface LabelRepository extends JpaRepository<Label, Long> {

    // SELECT * FROM label WHERE name = ?
    List<Label> findByName(String name);

    // SELECT * FROM label WHERE name = ? AND active = ?
    List<Label> findByNameAndActive(String name, boolean active);

    // SELECT * FROM label WHERE name LIKE ?
    List<Label> findByNameContaining(String keyword);

    // SELECT COUNT(*) FROM label WHERE active = ?
    long countByActive(boolean active);

    // DELETE FROM label WHERE name = ?
    void deleteByName(String name);
}

The naming convention is: findBy / countBy / deleteBy + field names + keywords like And, Or, Containing, Between, OrderBy...

Method name keywordSQL equivalent
findByNameWHERE name = ?
findByNameContainingWHERE name LIKE %?%
findByAgeGreaterThanWHERE age > ?
findByActiveTrueWHERE active = true
findByNameOrderByCreatedAtDescWHERE name = ? ORDER BY created_at DESC
findTop3ByOrderByCreatedAtDescORDER BY created_at DESC LIMIT 3

When Derived Queries Aren't Enough — @Query

For complex queries, use @Query with JPQL (or native SQL):

@Query("SELECT l FROM Label l WHERE l.genre = :genre AND l.active = true ORDER BY l.createdAt DESC")
List<Label> findActiveByGenre(@Param("genre") String genre);

// Or native SQL if you need MySQL-specific syntax:
@Query(value = "SELECT * FROM label WHERE MATCH(name) AGAINST (:term IN BOOLEAN MODE)",
       nativeQuery = true)
List<Label> fullTextSearch(@Param("term") String term);

🔍 LabelRadar Example — Full Entity + Repository

@Entity
@Table(name = "labels")
public class Label {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    private String genre;

    @Column(name = "created_at")
    private LocalDateTime createdAt;

    // getters + setters
}

@Repository
public interface LabelRepository extends JpaRepository<Label, Long> {

    List<Label> findByGenre(String genre);

    List<Label> findTop10ByOrderByCreatedAtDesc();

    @Query("SELECT l FROM Label l WHERE LOWER(l.name) LIKE LOWER(CONCAT('%', :q, '%'))")
    List<Label> searchByName(@Param("q") String q);
}
⚠️ MySQL 5.7 gotcha: spring.jpa.hibernate.ddl-auto=update will try to auto-create/alter tables. In production, set it to validate or none and use Flyway/Liquibase for migrations instead. This is especially important with MySQL 5.7 which has stricter default SQL mode.
